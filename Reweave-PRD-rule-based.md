## 0. 这份文档的核心论点

**编辑一份 HTML，工具不需要"理解"它，只需要把它切成一棵准确、可导航的组件层级树。**

"这个块是 hero 还是 cta"这类语义理解，对编辑功能毫无贡献。你复制一张卡片时，工具需要知道的是"这是一个可复制的重复单元、它的边界到哪里"——而不是它叫什么。把"理解"从地基里彻底拿掉之后，整个产品换来一组极其值钱的属性：

> **零延迟 · 离线可用 · 零调用成本 · 隐私无忧 · 行为完全可预测。**

对一个你打算天天用的 VS Code 插件，这组属性的价值远大于"组件名字更好看"。这份 PRD 把编辑内核做成 100% 纯规则；任何模型能力都是可选的、旁路的、默认关闭的增强，关掉不影响任何核心功能。

## 1. 一句话定位

**Reweave 用纯规则把任意 HTML——尤其是 AI 生成的单文件 HTML——解析成一棵可导航的组件树，让你在渲染视图上直接编辑既有组件，全程离线、即时、可预测，编辑产物始终是干净的 HTML。**

## 2. 为什么纯规则就够（核心论证）

要回答"纯规则行不行"，先把两件常被混为一谈的事拆开：

| | 切得准 | 叫得对 |
|---|---|---|
| 含义 | 把文档拆成正确的组件边界，认出哪些是同构的重复项 | 知道某个块在人类语义上"是什么"（hero / testimonial / pricing） |
| 谁需要它 | **所有编辑功能**：选区、复制、删除、移动、列表项同步 | 仅两处：UI 友好标签、组件库分类 |
| 纯规则能力 | **强** | 中（高混淆 class 下退化为"区块 N"） |

**结论：编辑功能全部建立在"切得准"之上，与"叫得对"无关。** 而"切得准"恰恰是规则的主场——它是一个结构问题，不是语义问题。

### 2.1 规则的王牌：重复子树检测

真实文档里绝大多数"组件感"强的结构，本质都是**重复模式**：卡片网格、导航菜单、特性列表、价目表、FAQ、时间线、步骤条……识别它们不需要任何语义理解，只需要识别"这几个兄弟节点结构相同"。

算法朴素而稳健——给每个元素计算一个**结构指纹**（忽略文本内容，基于标签 + class 模式 + 子结构递归），在兄弟序列里找指纹相同的连续段，即为一个重复组：容器是列表组件，每个成员是可复制 / 可同步的列表项。

```
fingerprint(node):
    if node is text/leaf: return hash(tag, classPattern)
    childSig = concat( fingerprint(c) for c in node.children )
    return hash(tag, classPattern, childSig)

detectRepeats(parent):
    groups = group consecutive siblings by fingerprint
    for g in groups where g.count >= 2:
        mark g as RepeatGroup(container=parent, items=g)
```

这一个信号就覆盖了用户心智中大半的"组件"，且零延迟、零歧义、可离线。

### 2.2 另外三类确定性信号

- **布局信号**：读 computed style，`display:grid/flex` 容器的直接子项即并列组件。视觉边界常比 DOM 嵌套更贴近人的认知。
- **语义标签**：`header / nav / main / section / article / aside / footer / figure / table / form` 命中即极准。
- **class 模式**：`.card / .hero / .btn / .cta / .container` 等约定俗成的命名。

> **目标场景的隐藏利好**：AI 生成的单文件 HTML 特别爱用语义化 class（训练数据就长这样），不像生产环境那种 `.css-1a2b3c` 混淆类名。所以在 Reweave 的核心场景里，class 信号意外地强——这正是纯规则路线在此场景成立的关键前提。

### 2.3 粒度歧义不是 bug，是交互职责

一张卡片里有图标 + 标题 + 描述，"组件"是整张卡还是标题？**规则给不出唯一答案，而且它不该给。** 这个歧义由交互解决：点一下选中最近一级，点"选父级"逐级向上扩。规则只需保证层级树是准的、可逐级跳的。

这比"让模型猜你要哪个粒度"更快、更可控、更可预测。而且**规则切错了也不致命**——用户拖一下边界就纠正了，纠正还能沉淀为规则补丁。

## 3. 背景与问题

AI 生成 HTML 的能力已经溢出，"可编辑性"严重滞后。根因不是"HTML 难"，而是**呈现复杂度被内联进了内容文档自身**：CSS、布局、组件结构全塞在文档里，内容被结构噪声淹没，导致"看到的渲染效果"和"要改的源码"差距巨大。

现有方案都不解决这个问题：

| 方案 | 为什么不够 |
|---|---|
| 飞书 / Notion | 固定块模型框死表现力，且吃不进外部 HTML |
| Webflow / 建站工具 | 锁定私有格式，无法编辑既有的任意 HTML，产物非干净 HTML |
| 直接改源码 | 正是要解决的痛点本身 |
| 让 AI 重新生成 | 改一处牵全身，不可控、丢失人对细节的把控 |
| 通用 DOM 块编辑器 | 不识别组件结构，退化成"能点的飞书" |

Reweave 的答案：**用规则把噪声折叠成可导航的组件树**，让人面对组件而非裸 DOM；导出时还原为标准 HTML。

## 4. 目标用户与场景

### 4.1 画像
- **P0 · AI 协作的技术内容创作者**：用 AI 生成 HTML 报告 / 文档 / 落地页，需高频精修。
- **P1 · 前端 / 全栈开发者**：调 AI 草稿或设计稿 HTML 的内容与样式，不想在渲染和源码间反复跳。
- **P2 · 懂点 HTML 的产品 / 运营**：维护营销页、技术博客、产品文档。

### 4.2 核心旅程
```
让 AI 生成 report.html
   → 右键 Open in Reweave
   → 规则即时解析：识别 hero、卡片×3（重复组）、CTA、脚注，层级树就绪
   → 渲染视图上：点选卡片 → 改文字 / 调主色 / 复制一份 / 删掉
   → 从组件库拖入引用块，或把改好的卡片"提取为可复用组件"
   → Ctrl+S → 最小 diff 写回源文件，git 改动清晰
全程无网络、无等待、无模型调用。
```

## 5. 设计原则

1. **工具导航结构，人理解语义。** 编辑内核只产出可导航的组件树，绝不试图"理解"文档含义。
2. **确定性是特性，不是妥协。** 同一份 HTML 永远解析出同一棵树；行为可预测、可调试、可复现。
3. **源文件是唯一真相。** 组件树、渲染视图都是源文件的投影，与 VS Code 文件模型 / git 一致。
4. **最小惊讶的回写。** 编辑只改它该改的源码区间，未触碰部分原样保留（含缩进、注释）。
5. **模型能力一律旁路。** 任何 LLM 增强都可选、默认关、关掉不影响核心；编辑器本体零模型依赖。

## 6. 核心概念

| 术语 | 定义 |
|---|---|
| **组件树（Component Tree / IR）** | 纯规则产出的中间表示。每节点：结构类型、DOM 范围、源码映射、可编辑文本槽、可调属性。 |
| **结构指纹（Structural Fingerprint）** | 忽略文本、基于标签 + class 模式 + 子结构递归的哈希。重复子树检测的基础。 |
| **重复组（Repeat Group）** | 指纹相同的连续兄弟序列。容器为列表组件，成员为可复制 / 可同步的列表项。 |
| **源码映射（Source Map）** | DOM 节点 ↔ 源码字节区间的双向映射，干净回写与增量同步的基础。 |
| **属性抽象（Property Abstraction）** | 从 computed style 反推出的可操作面板项（颜色 / 间距 / 字号 / 链接 / 图片源）。 |
| **结构命名（Structural Label）** | 纯规则给出的"够用"名称：语义标签名 / class 映射 / 序号（如"卡片项·2""区块 3"）。 |

## 7. 功能需求

### F1 · 规则识别引擎（核心，纯规则）

无网络、无模型、毫秒级。由四类信号叠加，重复子树检测为主干。

**Layer 0 — 解析与归一化**
- HTML → DOM AST，**为每节点记录源码区间**，建立 Source Map。
- 抽取样式来源：内联 `style`、文档内 `<style>`、同目录外链 CSS，计算 computed style。
- 优雅处理 AI 生成 HTML 的现实：内联样式为主、夹带 `<script>`、外部资源、不规范标签。

**Layer 1 — 结构切分（四信号）**
1. **重复子树检测**（主干）：识别重复组 → 列表组件 + 列表项。
2. **布局信号**：grid/flex 容器的并列子项。
3. **语义标签**：HTML5 结构标签。
4. **class 模式**：约定俗成的组件类名。
- 四信号融合为一棵组件树，**毫秒级、确定性、可离线**。

**结构命名（够用即可）**
- 优先级：语义标签 → 已知 class 映射 → 重复组序号 → 通用"区块 N"。
- 明确接受：命名不如语义模型漂亮，但对"选哪个组件"这个唯一用途完全够用。

> 引擎契约：解析永远成功、永远即时、永远可复现。没有"等模型返回"，没有"这次和上次切得不一样"。

#### IR Schema（示意）
```jsonc
{
  "id": "cmp_a1b2",
  "kind": "repeat-item",        // 结构类型（非语义）：block/repeat-group/repeat-item/leaf
  "label": "卡片项·1",           // 结构命名，纯规则
  "source": { "start": 1840, "end": 2103 },
  "domPath": "body>div.wrap>section:nth(2)>div.cards>div.card:nth(1)",
  "repeatGroupId": "grp_77",    // 属于哪个重复组 → 支持复制/同步
  "editable": {
    "text": ["h3", "p"],
    "props": [
      { "key": "accent", "kind": "color", "value": "#5b8cff", "binds": "border-color" },
      { "key": "padding", "kind": "spacing", "value": "24px" }
    ]
  }
}
```

### F2 · 可视化编辑画布
承载于 Webview，文档跑在内嵌 iframe 做硬隔离，编辑层叠加其上。
- **悬停高亮**：实时显示组件边界 + 结构命名。
- **点选 + 层级导航**：选最近组件，选父级 / 选子级逐级调整粒度。
- **内联文本编辑**：双击文本槽改写，粘贴清洗为纯文本。
- **结构操作**：复制、删除、上 / 下移、拖拽重排（拖拽 V1）。
- **画布工具**：缩放、响应式断点预览、大纲面板（组件树点选定位）。

### F3 · 属性与样式抽象层（本就是纯规则）
从 computed style 反推绑定关系，让用户**调属性而非写 CSS**：
- 颜色（主色 / 背景 / 文字，吸附文档调色板）、间距、排版、资源（图片 src / 链接 href）。
- **逃生舱**：高级用户可直接编辑该组件原始 CSS，同样走 Source Map 精确回写。
- 反推有歧义时（一个 computed 值可能来自多条声明），面板标注不确定并提供逃生舱兜底。

### F4 · 结构编辑与组件提取
- **提取为可复用组件**：选中任意结构 → 命名 → 存入组件库，把一次性 DOM 升格为资产。
- **列表项同步编辑**：对重复组成员，支持"改一个同步全部"（如统一所有卡片内边距）。重复检测已经提供了同步所需的全部信息，零额外智能。
- **样式自包含选项**：提取 / 插入时可选内联化依赖样式，保证组件插入任意文档呈现正确。

### F5 · 组件库与复用
- 内置模板组件（标题段 / 卡片 / 引用块 / callout / 栅格 / CTA）+ 用户提取的组件。
- 以工作区文件形式持久化，可入 git、可团队共享，不依赖云端账号。

### F6 · 源码双向同步
**源文件 = 唯一真相，IR 是投影。**
- **视图→源码**：每个编辑操作生成结构化 mutation（insert/delete/move/setText/setAttr），经 Source Map 映射为**最小 patch**，仅改目标区间。
- **源码→视图**：FS Watcher 监听变更（含外部编辑 / git），增量重解析，局部刷新，保持选区与滚动。
- **Split view**：左源码右渲染，双向定位（点组件高亮源码，反之亦然）。
- **撤销 / 重做**：复用 VS Code 原生文档历史，不自建快照栈，git 友好。

### F7 · 语义命名增强（可选旁路，默认关闭）
唯一会用到模型的地方，且**完全可选**：
- 当文档使用高混淆 class、结构命名退化为"区块 N"时，用户可手动触发一次语义命名，让模型为组件起人话名字。
- 默认关闭；开启需用户自带 key；只发送结构骨架（剥离正文）；失败静默回退到结构命名。
- **它只改 UI 上显示的标签文字，不参与任何编辑逻辑、不写回源码。** 关掉它，产品功能完整无损。

### Future · 自然语言编辑（独立功能线，非地基）
"把这张卡改成警告色、间距收紧"——把意图翻译成一组 mutation。
- 明确定位：这是**另一条独立功能线**，建立在 F1–F6 的 `selectComponent` / `applyMutation` 原语之上，**不是编辑器的地基**。
- 编辑器没有它照常完整可用；有它则多一种输入方式。作为远期探索，不进核心版本。

## 8. 技术架构（纯规则，简化）

```
┌──────────────────────────── VS Code ────────────────────────────┐
│  Extension Host (Node)                  Webview (沙箱)            │
│  ┌────────────────────────┐             ┌──────────────────────┐ │
│  │ 文件 I/O & FS Watcher   │◀──保存/监听─▶│ 编辑画布 (iframe 隔离)│ │
│  │ HTML 解析 + Source Map  │             │ 悬停/选取/工具条      │ │
│  │ 规则识别引擎 (Worker)   │◀──消息通道──▶│ 属性面板 / 组件库 UI  │ │
│  │ 组件库持久化            │             │ 大纲 / 响应式预览     │ │
│  │ Mutation → 最小 Patch   │             └──────────────────────┘ │
│  └────────────────────────┘                                       │
│                                                                    │
│  注：无 LLM 网关、无外部网络依赖、无隐私数据边界问题。              │
│  F7 语义命名为可选旁路，不在核心数据流内。                          │
└──────────────────────────────────────────────────────────────────┘
```

**数据流（单向真相）**
```
用户操作 → Webview 发 mutation
  → Host 校验 + Source Map 算最小 patch
  → 写源文件（VS Code 编辑 API，纳入 undo/git）
  → FS 变更触发增量重解析（纯规则，即时）
  → IR diff 推回 Webview，局部刷新
```
重计算（结构指纹、重复检测）放 Worker，主线程不阻塞。撤销复用 VS Code 文档历史。

## 9. 关键技术决策

| 决策点 | 选择 | 权衡 |
|---|---|---|
| 编辑内核 | 纯规则，零模型 | 即时/离线/可预测/隐私无忧；代价是命名不如语义模型漂亮 |
| 识别主干 | 重复子树检测 + 三类信号 | 覆盖大半真实组件，零歧义；扁平无规律 DOM 是已知盲区 |
| 粒度歧义 | 交给交互（选父/选子） | 快且可控；不试图自动猜测用户意图 |
| Source of Truth | 源文件 | git/VSCode 一致、可共存手写源码；需维护精确 Source Map |
| 回写 | 最小 patch | diff 干净可审；mutation→源码映射实现成本高 |
| 模型能力 | 旁路、可选、默认关 | 核心零依赖；语义命名作为锦上添花 |

## 10. 非功能需求
- **性能**：万级节点下结构切分 < 200ms，编辑回写 < 100ms。增量解析 + 视图虚拟化 + Worker。
- **兼容性**：覆盖主流 AI 生成 HTML 形态，优雅处理 `<script>` / 外部资源 / 不规范标签。
- **离线**：核心功能 100% 离线可用，安装即用，无需任何配置或登录。
- **隐私**：编辑内核不发送任何数据；唯一可选的 F7 仅在用户主动开启时发送结构骨架。
- **可访问性**：键盘可达、焦点可见、尊重 reduced-motion。
- **可靠性**：解析永远成功（最坏退化为单一根块也可编辑），回写前校验 Source Map。

## 13. 风险与开放问题（纯规则的真实风险）

1. **扁平无规律 DOM**：一堆同级 div、仅靠 margin 制造视觉分块、无 class 无语义标签——重复检测和布局信号都失效，退化为粗块。→ 缓解：纯视觉聚类的启发式（基于 bounding box 间距）作为补充信号；用户手动框定边界并沉淀为规则补丁。
2. **结构指纹的松紧度**：指纹太严会漏掉"结构相似但有细微差异"的列表项，太松会错并不相关的块。→ 需要一个可调的相似度阈值，并在校正中学习。
3. **Source Map 在连续结构编辑后的稳定性**：映射是否漂移。→ 严格的 mutation 不变量 + 回写后校验。
4. **属性反推歧义**：computed 值溯源到多条声明。→ 面板标注不确定 + 逃生舱兜底。
5. **范围蔓延**：易滑向"全功能建站工具"。→ 守住定位：**编辑既有 HTML，不做从零建站**。

> 与上一版（语义/LLM 路线）的本质区别：本版把 LLM 相关的全部风险（成本、延迟、准确率天花板、隐私边界）从核心路径中**彻底移除**，换来一个更小、更稳、更可预测的系统。代价仅有一项——在高混淆 class 文档上，组件名字不够漂亮。对一个日用编辑工具，这是笔划算的交易。
